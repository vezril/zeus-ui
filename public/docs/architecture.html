OLYMPUS — Architecture

The Codex homelab system: the dedicated VLAN (tag 70, 10.20.0.0/24) that the k3s cluster and service constellation run on — network → hardware → cluster → services → GitOps. Scroll to zoom · drag to pan.
Apollo (apollo-storage): a GCS-inspired, event-sourced object store in Scala 3 + Pekko. CQRS write/read split, the pure core domain, the read-model projection, and the NFS blob store. Scroll to zoom · drag to pan.
HermesMQ (hermesmq): an event-sourced Pub/Sub broker in Scala 3 + Pekko. Topic & Subscription aggregates, projection-driven at-least-once delivery, leases/redelivery/dead-letter/TTL sweepers, REST + gRPC. Scroll to zoom · drag to pan.

Key facts

Modules domain (pure, shared) + server (Pekko runtime) + client (typed REST lib)
Aggregates TopicEntity & SubscriptionEntity · EventSourcedBehavior · cluster-sharded (one writer/id)
APIs REST :8080 (/v1) + gRPC :8081 — TopicAdmin + PubSub (Publish/Pull/Ack/Stream/Consume)
Delivery at-least-once · DeliveryProjection tails topic journal → fans out to every subscription
Leases pull leases a message (ack-deadline 30s); unacked → RedeliverySweeper redelivers, counts attempts
Dead-letter at max attempts → republished to dead-letter topic (x-delivery-attempts headers)
TTL expireTime = publishTime + TTL; TtlSweeper drops expired (not dead-lettered)
Journal pekko-persistence-jdbc → PostgreSQL · explicit JSON · snapshots + retention (bounded)
Read models topic_subscriptions · outstanding_leases · expiring_messages · subscription/topic stats → /metrics
Multi-tenant salted-SHA-256 API keys · <tenant>~<id> isolation · admin vs pub/consume scopes
Cluster Cluster Sharding + single cluster-wide ShardedDaemonProcess projections · Artery :25520 · split-brain resolver
